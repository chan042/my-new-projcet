# TopMemo 개발 계획서
- 버전: v1
- 프로젝트 형태: 신규 macOS 앱

## 1. 목표

TopMemo는 맥북 상단 메뉴막대에 상주하는 초경량 메모 앱이다. 사용자는 메뉴막대의 TopMemo 아이콘을 클릭한 즉시 아이콘 바로 아래로 내려오는 메모 UI에서 곧바로 타이핑할 수 있어야 한다.

핵심 목표는 다음 4가지다.

1. 클릭 후 즉시 입력 가능한 속도
2. 복잡한 기능 없이 저장, 삭제, 뒤로가기만 있는 단순성
3. 여러 메모를 빠르게 확인하고 새 메모를 추가할 수 있는 메뉴막대 중심 UX
4. 인터넷 배포가 가능한 `.dmg` 릴리스 파이프라인 확보

## 2. 요구사항 해석

### 2.1 필수 기능

1. 앱은 macOS 메뉴막대에 상주한다.
2. 메뉴막대 아이콘 클릭 시 아이콘 바로 아래에 메모 UI가 열린다.
3. 첫 실행 또는 저장된 메모가 없을 때는 즉시 타이핑 가능한 작은 메모장이 먼저 보인다.
4. 첫 저장 이후에는 저장된 메모들이 상단에 작은 미리보기 형태로 보인다.
5. 하단 `+` 버튼으로 새 메모 작성 화면으로 진입한다.
6. 메모 편집 화면에는 좌측 상단에 `뒤로가기`, 우측 상단에 `삭제`, `저장`만 있다.
7. 텍스트 색상은 `검정`, `빨강`, `파랑`, `초록` 4가지를 지원한다.
8. 기본 텍스트 색상은 `검정`이다.
9. 앱을 `.dmg` 형태로 패키징해 배포 가능해야 한다.

### 2.2 본 계획에서의 구체적 해석

요구사항에 일부 모호한 부분이 있으므로 아래처럼 해석해 개발한다.

1. 저장 이후의 "상단 미리보기"는 단일 메모가 아니라 저장된 메모 목록의 작은 카드 리스트로 처리한다.
2. 메모 상세 편집 화면에는 제목 없이 본문만 입력한다.
3. 메모는 로컬 저장이 기본이며, iCloud 동기화는 v1 범위에서 제외한다.
4. 메뉴막대 팝오버를 닫으면 자동 저장된다.
5. 메모 정렬은 최신 수정순으로 한다.

## 4. 권장 기술 선택

TopMemo는 네이티브 macOS 앱으로 개발하는 것이 가장 적합하다.

### 4.1 프레임워크

1. `SwiftUI`
   - UI 작성 속도가 빠르다.
   - 텍스트 편집, 상태 관리, 프리뷰 작성이 편하다.
2. `AppKit`
   - 메뉴막대 상태 아이템(`NSStatusItem`)과 팝오버(`NSPopover`) 제어를 위해 필요하다.
   - "아이콘 바로 아래로 내려오는" 동작을 더 정확히 제어할 수 있다.

### 4.2 저장 방식

v1에서는 `Application Support/TopMemo/notes.json` 형태의 로컬 JSON 저장을 권장한다.

이유:

1. 기능이 단순하다.
2. SwiftData/Core Data보다 구현 복잡도가 낮다.
3. 백업/마이그레이션 포인트를 직접 통제하기 쉽다.
4. `.dmg` 배포용 경량 앱과 궁합이 좋다.

### 4.3 최소 지원 OS

권장 최소 버전은 `macOS 13+`로 잡는다.

이유:

1. 최신 SwiftUI API 사용이 편하다.
2. 메뉴막대 기반 앱 구현 시 레이아웃 및 포커스 제어가 안정적이다.
3. 유지보수 비용을 낮출 수 있다.

필요하면 이후 `macOS 12`까지 하향 검토 가능하다.

## 5. UX 설계

## 5.1 주요 사용자 흐름

### 흐름 A: 첫 실행

1. 사용자가 앱 실행
2. 메뉴막대에 TopMemo 아이콘 표시
3. 아이콘 클릭
4. 빈 메모 편집 팝오버 즉시 노출
5. 커서가 본문 입력창에 자동 포커스
6. 사용자가 내용 입력
7. 색상 선택 가능, 기본값은 검정
8. `저장` 클릭
9. 팝오버는 리스트 뷰로 전환

### 흐름 B: 저장 이후

1. 사용자가 아이콘 클릭
2. 팝오버 상단에 저장된 메모 미리보기 리스트 표시
3. 각 미리보기는 1~3줄 요약 형태
4. 하단 고정 영역에 `+` 버튼 표시
5. 사용자가 미리보기를 누르면 편집 화면으로 진입
6. 사용자가 `+`를 누르면 새 메모 작성 화면으로 진입

### 흐름 C: 편집 중

1. 상단 바에는 `뒤로가기`, `삭제`, `저장`만 노출
2. 본문 하단 또는 상단 하위 영역에 색상 팔레트 표시
3. `저장` 시 리스트 반영
4. `삭제` 시 확인 절차 후 리스트에서 제거
5. `뒤로가기`는 저장하지 않은 변경사항이 있으면 경고 모달 표시 여부를 결정해야 함

## 5.2 화면 구성

### A. 메뉴막대 상태

1. 메뉴막대에 심볼 기반 단색 아이콘 표시
2. 클릭 시 팝오버 토글
3. 외부 클릭 시 팝오버 닫힘

### B. 리스트 화면

권장 크기:

- 폭 `360`
- 높이 `420` 내외

구성:

1. 상단: 앱 이름이 들어간 매우 얇은 헤더
2. 중단: 메모 미리보기 리스트
3. 하단: `+ 새 메모` 버튼

각 카드 표시 요소:

1. 본문 일부
2. 색상 표시 힌트
3. 수정 시간(작게)

### C. 편집 화면

권장 크기:

- 폭 `360`
- 높이 `300 ~ 420`

구성:

1. 상단 헤더
   - `뒤로가기`
   - `삭제`
   - `저장`
2. 본문 입력 영역
3. 색상 팔레트

## 5.3 상호작용 세부 규칙

1. 팝오버가 열리면 가장 먼저 키보드 포커스가 텍스트 입력창에 가야 한다.
2. `Esc`는 가능하면 팝오버 닫기로 동작시킨다.
3. `Cmd+S`는 저장 단축키로 연결한다.
4. `Cmd+N`은 새 메모 작성으로 연결할 수 있다.
5. 미리보기 텍스트는 줄 수를 제한하고 말줄임 처리한다.
6. 삭제는 실수 방지를 위해 확인 대화상자를 권장한다.

## 6. 정보 구조 및 데이터 모델

### 6.1 핵심 모델

```swift
struct MemoItem: Identifiable, Codable, Equatable {
    let id: UUID
    var content: String
    var color: MemoColor
    let createdAt: Date
    var updatedAt: Date
}

enum MemoColor: String, Codable, CaseIterable {
    case black
    case red
    case blue
    case green
}
```

### 6.2 뷰 상태

```swift
enum TopMemoRoute {
    case emptyComposer
    case memoList
    case editor(memoID: UUID?)
}
```

### 6.3 저장 정책

1. 앱 실행 시 `notes.json` 로드
2. 저장 버튼 클릭 시 전체 배열을 디스크에 반영
3. 삭제 시 즉시 디스크 반영
4. 정렬 기준은 `updatedAt DESC`

## 7. 권장 아키텍처

메뉴막대 앱 특성상 SwiftUI만으로 끝내기보다 AppKit 브리지가 있는 얇은 계층 구조가 적합하다.

### 7.1 구조 개요

1. `App`
   - 앱 생명주기 진입점
2. `StatusBar`
   - `NSStatusItem` 생성 및 클릭 이벤트 처리
3. `Popover`
   - `NSPopover` 열기/닫기 및 위치 제어
4. `Domain`
   - 메모 모델, 색상 enum
5. `Persistence`
   - JSON 파일 읽기/쓰기
6. `Features/Notes`
   - 리스트 화면, 편집 화면, 상태 관리

## 8. 구현 단계별 계획

## 8.1 1단계: 프로젝트 초기화

목표:

1. Xcode macOS App 프로젝트 생성
2. 앱 번들 ID, 팀 설정, 배포 타깃 정리
3. 기본 메뉴막대 앱 골격 구성

작업:

1. Xcode 프로젝트 생성
2. `LSUIElement` 설정으로 Dock 아이콘 없는 메뉴막대 앱 동작 검토
3. 자산 카탈로그에 임시 메뉴막대 아이콘 추가
4. 빈 팝오버 표시까지 연결

완료 기준:

1. 앱 실행 시 메뉴막대 아이콘이 보인다.
2. 아이콘 클릭 시 빈 팝오버가 열리고 닫힌다.

## 8.2 2단계: 데이터 계층 구현

목표:

1. 메모 모델과 로컬 저장 기능 구현
2. 앱 재실행 후 데이터 복원

작업:

1. `MemoItem`, `MemoColor` 구현
2. `NotesStore` 구현
3. `Application Support` 경로 생성 로직 구현
4. JSON 인코딩/디코딩 및 오류 처리 구현

완료 기준:

1. 샘플 데이터 저장/로드가 된다.
2. 앱 재실행 후 메모가 유지된다.

## 8.3 3단계: 첫 실행용 편집 화면 구현

목표:

1. 저장된 메모가 없을 때 즉시 입력 가능한 화면 제공

작업:

1. 빈 메모 편집 뷰 구현
2. 최초 포커스 자동 부여
3. 기본 텍스트 색상 검정 적용
4. 저장 버튼 동작 연결

완료 기준:

1. 첫 클릭 후 바로 타이핑 가능하다.
2. 저장 시 메모가 실제 저장된다.

## 8.4 4단계: 저장 이후 리스트 화면 구현

목표:

1. 저장된 메모 미리보기 목록과 새 메모 진입 흐름 구현

작업:

1. 메모 리스트 카드 UI 구현
2. 수정일 표시
3. `+` 버튼 구현
4. 카드 클릭 시 편집 화면 이동 구현

완료 기준:

1. 저장 이후에는 리스트 화면이 기본 진입점이 된다.
2. `+` 버튼으로 새 메모 작성이 가능하다.

## 8.5 5단계: 편집/삭제/뒤로가기 완성

목표:

1. 요구사항의 최소 조작만으로 모든 메모 수명주기를 처리

작업:

1. 기존 메모 편집 기능 구현
2. 삭제 기능 구현
3. 뒤로가기 처리 구현
4. 저장하지 않은 변경 감지 여부 결정 및 반영

완료 기준:

1. 새 메모 생성, 기존 메모 수정, 삭제가 모두 된다.
2. 뒤로가기가 리스트/초기 화면과 자연스럽게 연결된다.

## 8.6 6단계: 팔레트 및 시각 정리

목표:

1. 4색 팔레트를 명확하고 단순하게 제공

작업:

1. 검정/빨강/파랑/초록 팔레트 버튼 UI
2. 현재 선택 색상 표시
3. 미리보기 카드에도 색상 힌트 반영
4. 기본 타이포/간격/모서리/그림자 정리

완료 기준:

1. 모든 메모는 지정된 4색 중 하나를 가진다.
2. 기본 색상은 항상 검정으로 시작한다.

## 8.7 7단계: 안정화 및 품질 점검

목표:

1. 메뉴막대 앱 특유의 포커스/팝오버 버그를 줄인다.

작업:

1. 팝오버 열고 닫기 반복 테스트
2. 입력 포커스 누락 여부 확인
3. 외부 클릭 종료 동작 확인
4. 빈 메모 저장 제한 여부 결정
5. 긴 텍스트 렌더링 및 스크롤 테스트

완료 기준:

1. 주요 상호작용에서 UI 깨짐이 없다.
2. 저장/삭제/뒤로가기 흐름이 안정적이다.

## 8.8 8단계: 배포 준비

목표:

1. 외부 사용자가 설치 가능한 `.dmg` 산출물 확보

작업:

1. Release 빌드 설정 정리
2. 앱 서명(Signing)
3. Apple notarization
4. `.app`를 포함한 `.dmg` 생성 스크립트 작성
5. 깨끗한 환경에서 설치 테스트

완료 기준:

1. `.dmg`를 열어 앱을 `/Applications`로 드래그할 수 있다.
2. Gatekeeper 경고 없이 실행 가능하도록 서명/노타라이즈가 완료된다.

## 9. 테스트 계획

## 9.1 수동 테스트

반드시 확인할 시나리오:

1. 앱 최초 실행 후 첫 메모 저장
2. 두 번째 메모 생성
3. 기존 메모 수정 후 저장
4. 메모 삭제
5. 메모 색상 변경 후 리스트 반영
6. 앱 종료 후 재실행 데이터 유지
7. 팝오버 외부 클릭 종료
8. 매우 긴 텍스트 입력
9. 빈 텍스트 저장 동작 정책 확인

## 9.2 자동 테스트

v1에서는 최소한 아래를 권장한다.

1. `NotesStore` 단위 테스트
2. JSON 직렬화/역직렬화 테스트
3. 메모 정렬 테스트
4. 삭제 후 저장 반영 테스트

UI 테스트는 최소화하되, 핵심 플로우 1~2개는 추후 추가 검토한다.

## 10. `.dmg` 배포 계획

## 10.1 배포 산출물

1. `TopMemo.app`
2. `TopMemo.dmg`
3. 릴리스 노트
4. 필요 시 SHA256 체크섬

## 10.2 배포 절차

1. Release Archive 생성
2. `Developer ID Application` 인증서로 서명
3. `notarytool`로 notarization 제출
4. notarization 통과 후 staple 적용
5. `.app`를 담은 `.dmg` 생성
6. 다운로드 링크 업로드

## 10.3 구현 시 유의점

1. 인터넷 배포용 macOS 앱은 서명과 notarization이 사실상 필수다.
2. 미서명 `.dmg`는 사용자 경험이 매우 나빠질 수 있다.
3. 배포 자동화는 `make-dmg.sh` 또는 CI 스크립트로 분리한다.

## 11. 예상 리스크와 대응

### 리스크 1: 메뉴막대 팝오버 포커스 이슈

위험:

1. 팝오버는 열렸지만 텍스트 커서가 자동으로 들어가지 않을 수 있다.

대응:

1. `AppKit` 브리지로 첫 responder 제어
2. 필요 시 `NSTextView` 래핑 검토

### 리스크 2: SwiftUI TextEditor 커스터마이징 한계

위험:

1. 텍스트 색상, 포커스, 여백 제어가 기대만큼 세밀하지 않을 수 있다.

대응:

1. 우선 `TextEditor`로 구현
2. 문제 발생 시 `NSTextView` 기반 커스텀 래퍼로 교체

### 리스크 3: 삭제/뒤로가기에서 데이터 손실 혼동

위험:

1. 저장하지 않은 변경사항이 사라질 수 있다.

대응:

1. 변경 감지 플래그 사용
2. 필요 시 간단한 확인 대화상자 제공

### 리스크 4: 빈 메모 처리 정책

위험:

1. 빈 메모를 허용하면 리스트 품질이 낮아질 수 있다.

대응:

1. 공백만 있는 메모는 저장 금지 권장
2. 저장 버튼 비활성화 또는 저장 시 무시

## 12. 완료 기준

아래 조건을 모두 만족하면 v1 완료로 본다.

1. 메뉴막대에서 TopMemo 아이콘이 항상 정상 표시된다.
2. 클릭 시 아이콘 아래에 팝오버가 자연스럽게 열린다.
3. 첫 실행 시 바로 입력 가능한 빈 메모장이 보인다.
4. 저장 후에는 메모 미리보기 목록과 `+` 버튼이 보인다.
5. 메모 편집 화면에는 `뒤로가기`, `삭제`, `저장`만 있다.
6. 텍스트 색상 4종이 모두 동작한다.
7. 앱 재실행 후 데이터가 유지된다.
8. `.dmg`를 통해 설치 가능한 Release 빌드가 생성된다.

## 13. 바로 다음 실행 순서

계획서 기준으로 실제 개발은 다음 순서로 진행하는 것이 가장 효율적이다.

1. Xcode macOS 메뉴막대 앱 프로젝트 생성
2. 메뉴막대 아이콘 + 팝오버 골격 구현
3. 메모 저장 모델 및 JSON 저장소 구현
4. 첫 실행 편집 화면 구현
5. 저장 이후 리스트 화면 구현
6. 편집/삭제/뒤로가기/팔레트 마감
7. 테스트 및 `.dmg` 배포 자동화

## 14. 권장 의사결정 메모

개발 시작 전에 아래 3가지는 빠르게 확정하면 좋다.

1. 최소 지원 macOS 버전(`13+` 권장)
2. Dock 아이콘 완전 숨김 여부
3. 빈 메모 저장 금지 여부

이 3가지만 확정되면 구현이 거의 흔들리지 않는다.
